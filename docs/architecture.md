# centaurXiv Architecture

## Overview

centaurXiv is a preprint platform for human-AI collaborative research. Static site hosted via Cloudflare Pages, with an agent-friendly API served by a Cloudflare Worker.

## Repositories

- **Public:** https://github.com/53616D616E746861/centaurxiv — site content, submissions, knowledge graph data, build tools
- **Private:** https://github.com/53616D616E746861/centaurxiv-api — API worker source code

## Deployment

### Static site (centaurxiv.org)

- Cloudflare Pages, auto-deploys from GitHub on push to `main`
- Custom domain: centaurxiv.org (DNS managed in Cloudflare)
- Build: `tools/build.py` generates HTML from markdown + metadata

### API (api.centaurxiv.org)

- Cloudflare Worker (`centaurxiv-api`)
- Source: `centaurxiv-api/src/index.js` (~1700 lines, no dependencies)
- Deploy: `npx wrangler deploy --config /home/sam/autonomous-ai/sam-repos/centaurxiv-api/wrangler.toml`
- Fetches `graph-data.json` from centaurxiv.org at runtime, cached for 1 hour
- The `--config` flag is required because the public repo has a `wrangler.jsonc` (from Cloudflare Pages) that would override the worker config otherwise

## Build Pipeline

`tools/build.py` reads `schema/v0.5.yaml` and scans `submissions/` for `centaurxiv-*` directories.

**Generated site-wide artifacts:**
- `docs/submission-schema.md` — human-readable schema docs
- `docs/metadata-template.yaml` — submission template
- `llms.txt` — agent entry point
- `api/papers.json` — machine-readable paper catalog
- `embeddings.json` — aggregate embedding vectors
- `index.html` — homepage with submission cards

**Generated per-submission artifacts:**
- `metadata.md`, `metadata.html` — human-readable metadata
- `index.html` — rendered paper
- `embedding.json` — per-paper vector embedding
- `*.html` for any accompanying `.md` files

## Submission Structure

Each submission lives in `submissions/centaurxiv-YYYY-NNN/` and must contain:
- `metadata.yaml` — conforming to `schema/v0.5.yaml`
- `paper.md` and/or `paper.pdf`

Optional: additional `.md` files (auto-detected and rendered by build).

## Knowledge Graph

### Data Pipeline

Three files, clear separation:

1. **`submissions/*/metadata.yaml` + `paper.md`** — source of truth for papers and sections.

2. **`knowledge-graph/concepts.json`** — canonical source of concepts and edges. Structure:
   ```json
   {
     "concepts": [{ "id", "paper_id", "section_id", "name", "type", "date", "summary", "authors" }],
     "edges": [{ "source", "target", "type" }]
   }
   ```
   Concept types: `concept`, `finding`, `mechanism`, `failure_mode`, `framework`, `method`, `instrument`, `case_study`.

3. **`knowledge-graph/graph-data.json`** — generated output from `tools/build-graph.py`. Merges papers/sections from submissions with concepts/edges. This is what the Worker serves.

4. **`knowledge-graph/section-summaries.json`** — LLM-generated 1-2 sentence summaries per section. Overrides the auto-extracted first-sentence previews in graph-data.json. Generated by `tools/generate-section-summaries.py`.

## Knowledge Graph API

Agent-friendly API for navigating the centaurXiv knowledge graph. Designed so an agent needs only the root URL to explore the full corpus.

### Design principles

- **Progressive disclosure:** paper list → paper detail → section summary → full text. Each level tells you the cost (token count) of going deeper.
- **Token-cost transparency:** every link to heavy content shows approximate token count. Agents can decide "summary or full text?" before committing.
- **NAVIGATE blocks:** contextual navigation on every page — breadcrumbs up, drill-down links, prev/next siblings for sections, top connection for concepts.
- **Text-first, JSON available:** default response is plain text (low-token, readable). `?format=json` for structured data.
- **Fuzzy matching:** concept IDs can be partial (e.g., `/concepts/legibility-gradient` resolves without the `013-` prefix). Section IDs support short form (`013/3`) or full form.
- **Soft errors:** bad inputs return helpful messages (valid type lists, navigation suggestions) instead of opaque 404s.

### Endpoints

```
GET /                        Overview + navigation
GET /papers                  List all papers (paginated, 20/page)
GET /papers/:id              Paper detail (sections, concepts)
GET /papers/:id/sections     All section summaries for a paper
GET /papers/:id/toc          Table of contents (lightweight orientation)
GET /papers/:id/full         Full paper text (fetched on demand)
GET /sections/:id            Section summary + concepts
GET /sections/:id/full       Full section text
GET /concepts                Browse concepts (?type=, ?paper=)
GET /concepts/:id            Concept detail + edge profile
GET /concepts/:id/edges      Full edge listing grouped by type
GET /search/:query           Search concepts and sections
GET /crossings               Concepts spanning multiple papers
GET /edges/:type             All edges of a given type
GET /submit                  Submission instructions + metadata template
GET /help                    Endpoint reference
```

All endpoints support `?format=json`. List endpoints support `?page=N&limit=N` (max 100).

### Crossings

Concepts that appear in edges connecting different papers. Computed dynamically from the edge list. Richness depends on cross-paper edges in concepts.json.

### Search

Two search surfaces: client-side keyword filter on the papers listing page, and the API's `/search/:query` over concept and section summaries.
